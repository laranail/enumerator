# v0.3.0 scope — laranail/enumerator

Forward-looking planning doc, drafted 2026-05-15 immediately after
v0.2.1 shipped. Pulls the deferred items from
[`feature-gap.md`](../research/feature-gap.md) plus the lessons
learned during the v0.2.0 / v0.2.1 ship cycle.

This is a **scope draft**, not a completion plan. A future session
should run a fresh Phase 1 audit (per
[`audit-remediate.md`](../_private/audit-remediate.md)) before locking
the PR sequence. Numbers and acceptance criteria below are
best-effort estimates from this seat.

**Constraints carried forward from v0.1.x / v0.2.x:**

- Backwards compatibility within v1 is hard. No public-API removals
  or renames until v1.0.0 (see [ADR-0001](../decisions/0001-frame-as-v020-audit-not-pre-release.md)).
- The 21 + 9 baked decisions ([NEXT-SESSION.md](../_private/NEXT-SESSION.md)
  `D34..D64`) must not regress.
- PHP `^8.3` / Laravel `^13.0` floor stays.

## Lessons from v0.2.0 / v0.2.1 (apply in v0.3.0)

These three matter enough to call out at the top — they shape the
sequence below.

1. **Coverage-then-gate, not the other way around.** v0.2.0 raised
   `--min=80 → 85` ahead of measurement on the assumption that 38
   new tests would lift `src/` coverage. Most of those tests
   exercised Blade view rendering (excluded from `phpunit.xml`
   `<source>`), so `src/` stayed at 83.2%, CI red, v0.2.1 had to
   lower the gate back to 83. **Always measure first; raise the
   gate to match the new floor in a follow-up.**

2. **PHPStan baseline drift is silent.** `reportUnmatchedIgnoredErrors:
   false` is currently set so the baseline never fails the gate.
   That's been a useful shock-absorber across the PHP 8.3/8.4/8.5
   matrix, but it also means baseline entries can rot indefinitely.
   F-22 (below) addresses this.

3. **Tag push → release.yml auto-creates the GitHub release.** Don't
   `gh release create` after the push; it'll 422. Use `gh release
   edit` instead, feeding the extracted `[X.Y.Z]` CHANGELOG block as
   `--notes-file`. The Manual external steps in
   [`NEXT-SESSION.md`](../_private/NEXT-SESSION.md) capture this.

## Suggested ordering

For one cohesive v0.3.0 ship cycle, in order of dependency:

```
PR-α (baseline shrink, foundation)
   ↓
PR-β (coverage push)
   ↓
PR-γ (per-input wireModel)   ← natural follow-up to PR-F
PR-δ (multi-select listbox)  ← natural follow-up to PR-G
   ↓
PR-ε (coverage / mutation reporting)
   ↓
Release prep + tag v0.3.0
```

**PR-ζ + PR-η + PR-θ are nice-to-have.** If two of them slip to
v0.3.1 / v0.4.0, the v0.3.0 cycle is still a clean release.

## Open scope questions for the v0.3.0 audit-remediate pass

1. **Coverage target for v0.3.0 — 85% or 90%?** The current trajectory
   says 85% is reachable in 2-3 batches; 90% takes 4-5 and starts to
   feel like coverage-cargo-culting (the original assessment in
   [NEXT-SESSION.md](../_private/NEXT-SESSION.md) "The hard
   remaining 90% gap"). Decide before opening PR-β.

2. **Mutation testing on day 1, or after baseline?** `infection`
   integration matures across versions; a first-pass setup before
   PR-α's baseline shrink might noise-up the picture. After PR-α is
   cleaner.

3. **Browser-level CI test (PR-θ) — Playwright or Cypress?** Playwright
   has lighter CI overhead but a steeper Laravel-integration story.
   Cypress is heavier but better-documented for Laravel apps. Both
   add a `node_modules/` to the test rig. Worth a dedicated ADR.

4. **Continuous-deployment to Packagist via tag push?** Currently the
   Packagist hook would need to be wired manually (still in
   `NEXT-SESSION.md` "Manual external steps"). Once registered, every
   tag push auto-updates Packagist. Worth confirming before v0.3.0
   tag.
