# v0.3.0 scope — laranail/enumerator

Forward-looking planning doc, drafted 2026-05-15 immediately after
v0.2.1 shipped. Pulls the deferred items from
[`feature-gap.md`](../research/feature-gap.md) plus the lessons
learned during the v0.2.0 / v0.2.1 ship cycle.

This is a **scope draft**, not a completion plan. A future session
should run a fresh Phase 1 audit (per
[`audit-remediate.md`](../_private/audit-remediate.md)) before locking
the PR sequence. Numbers and acceptance criteria below are
best-effort estimates from this seat.

**Constraints carried forward from v0.1.x / v0.2.x:**

- Backwards compatibility within v1 is hard. No public-API removals
  or renames until v1.0.0 (see [ADR-0001](../decisions/0001-frame-as-v020-audit-not-pre-release.md)).
- The 21 + 9 baked decisions ([NEXT-SESSION.md](../_private/NEXT-SESSION.md)
  `D34..D64`) must not regress.
- PHP `^8.3` / Laravel `^13.0` floor stays.

## Lessons from v0.2.0 / v0.2.1 (apply in v0.3.0)

These three matter enough to call out at the top — they shape the
sequence below.

1. **Coverage-then-gate, not the other way around.** v0.2.0 raised
   `--min=80 → 85` ahead of measurement on the assumption that 38
   new tests would lift `src/` coverage. Most of those tests
   exercised Blade view rendering (excluded from `phpunit.xml`
   `<source>`), so `src/` stayed at 83.2%, CI red, v0.2.1 had to
   lower the gate back to 83. **Always measure first; raise the
   gate to match the new floor in a follow-up.**

2. **PHPStan baseline drift is silent.** `reportUnmatchedIgnoredErrors:
   false` is currently set so the baseline never fails the gate.
   That's been a useful shock-absorber across the PHP 8.3/8.4/8.5
   matrix, but it also means baseline entries can rot indefinitely.
   F-22 (below) addresses this.

3. **Tag push → release.yml auto-creates the GitHub release.** Don't
   `gh release create` after the push; it'll 422. Use `gh release
   edit` instead, feeding the extracted `[X.Y.Z]` CHANGELOG block as
   `--notes-file`. The Manual external steps in
   [`NEXT-SESSION.md`](../_private/NEXT-SESSION.md) capture this.

## PR-α tactical breakdown

Re-measured 2026-05-15 against HEAD `8370020` (the v0.3.0 scope-doc
commit). `phpstan-baseline.neon` is 413 KB / 7087 lines / **1181
unique error patterns** / 2072 total occurrences.

Top 10 error identifiers in the baseline (ranked by occurrence
count) — these are the high-leverage targets:

| # | Identifier | Occurrences | First-pass category | Notes |
|---|---|---:|---|---|
| 1 | `varTag.nativeType` | 225 | mostly **fixable** | `@var` annotations duplicating native PHP types. Remove or rewrite. |
| 2 | `function.alreadyNarrowedType` | 210 | mostly **fixable** | Defensive narrowing PHPStan can already prove. Remove the runtime check OR relax the docblock. |
| 3 | `return.type` | 143 | mixed | Some are real bugs (return type mismatch); some are generic-template noise. Triage per-file. |
| 4 | `argument.type` | 136 | mixed | Same triage. Tends to cluster in `Casts/` + `Eloquent/`. |
| 5 | `cast.string` | 74 | mostly **noise** | The `(string) $mixed` pattern in `HasAttributes::resolvedAttribute()`. Either widen the trait's docblock to declare `mixed` return, or push the cast site to a narrower scope. |
| 6 | `missingType.iterableValue` | 66 | mostly **fixable** | Array shape annotations missing. Add `@param array<...>` / `@return array<...>` annotations. |
| 7 | `method.nonObject` | 40 | mostly **real** | Calling a method on a possibly-null / mixed-typed value. Real bugs hiding here. Look at `MagicComparisonsTest` cluster. |
| 8 | `cast.int` | 40 | mostly **noise** | Same shape as `cast.string`. |
| 9 | `instanceof.alwaysTrue` | 39 | mostly **fixable** | Redundant runtime checks. Remove. |
| 10 | `staticMethod.notFound` | 28 | mostly **fixable** | Add `@method static` annotations on the relevant class. |

**Total addressable on this rank list:** 1001 of 2072 occurrences
(~48%). If even half of those collapse cleanly to code/doc fixes, the
baseline shrinks well past the 50% halve target.

Suggested PR-α-batch sequence:

1. **`varTag.nativeType` + `function.alreadyNarrowedType`** (435 occ).
   Mechanical removal; one focused batch.
2. **`missingType.iterableValue` + `instanceof.alwaysTrue` +
   `staticMethod.notFound`** (133 occ). Annotation-only or
   deletion-only fixes.
3. **`cast.string` + `cast.int`** (114 occ). Widen `HasAttributes`
   trait docblock if that's the root cause; if not, narrow per-site.
4. **`return.type` + `argument.type`** (279 occ). Triage per-file —
   highest variance, slowest. Some real bugs likely hiding here.
5. **`method.nonObject`** (40 occ). Likely-real defects — small
   batch but each one needs a code fix, not a docblock.

After steps 1-5, regenerate the baseline. If the count is below
600 unique patterns, flip `reportUnmatchedIgnoredErrors` to `true`
in `phpstan.neon` and watch the next CI run.

The 718 remaining patterns break down roughly as:

- **~150** in `tests/Fixtures/Enums/*` — each fixture instantiates the
  full trait stack, so per-fixture errors persist until the trait
  surface is itself silenced. Not high-leverage.
- **~150** in test files using Pest's `expect()->not->toContain()`
  against `Blade::render()` (typed as `string|null` by larastan).
  Same pattern across `AlpineLoaderTest`, `DropdownAlpineTest`,
  `LivewireAttributesTest`, etc. — each test file generates its own
  baseline entries. Fixable only by retyping `Blade::render` or
  casting `(string) $html` everywhere.
- **~100** scattered across `Console/*`, `Modules/StructuredOutput/*`,
  `Modules/Lighthouse/*`, `Support/AttributesCache.php` — these are
  one-off type-narrowing opportunities without trait-cascade leverage.
- **Residual `function.alreadyNarrowedType`** — 210 patterns deferred
  by Batch 1 because they're defensive guards for trait composition
  on heterogeneous hosts. Removing them breaks documented contracts
  (non-Eloquent hosts, headless-PHP fallback).
- Test files in `tests/Feature/AbstractEnumeratorClassTest.php` +
  `tests/Feature/DynamicEnumsTest.php` — `staticMethod.notFound` on
  AbstractEnumeratorClass `__callStatic` resolution. Needs `@method
  static` annotations on each fixture; mechanical but tedious.

### Why `reportUnmatchedIgnoredErrors: true` stays deferred

The scope doc's flip criterion ("below 600 unique patterns") was
not met. Three reasons not to flip yet:

1. **Cross-version drift.** Baseline regenerated against PHP 8.5
   locally. CI runs against 8.3 + 8.4 + 8.5; the 8.5-anchored baseline
   may have entries that don't fire on 8.3 or 8.4 (PHPStan's
   per-version inference differs). Flipping `true` would fail CI on
   the 8.3 / 8.4 jobs for every drift entry. The existing comment in
   `phpstan.neon:21` already documents this.
2. **The remaining 718 entries are dominated by Pest typing and
   trait-fixture composition** — neither cleanly resolves without a
   wider refactor of how the package's traits compose.
3. **The cost of the flip (false-positive CI failures) exceeds the
   benefit (catching baseline rot)** at this baseline size. Flip
   becomes worth it once the baseline is below ~300 entries.

### v0.3.x follow-up candidates

The PR-α residue motivates 2-3 follow-up sub-batches in v0.3.x:

- **PR-α-resid-1** — Test-file Blade::render typing fix.
  **TRIED 2026-05-15 — DEAD-END for this approach.** Two
  experiments, both failed:
  1. larastan stub at `phpstan-stubs/blade-facade.stub` declaring
     `Blade::render` as returning `string` not `string|null` — the
     stub had no effect on the baseline (stub loaded but the
     property.notFound errors still fired).
  2. Mechanical `(string) Blade::render(...)` cast at all 37
     `$html = Blade::render(...)` assignment sites. The cast
     correctly makes `$html: string` at that line, but Pest's
     `expect(mixed $value = null): Expectation` accepts the value
     as mixed and returns `Expectation<TValue>` where TValue is
     determined by the larastan Pest-integration stub, not the
     cast site. After the cast, phpstan reported
     `Expectation<mixed>::$not` instead of
     `Expectation<string|null>::$not` — different error message,
     same baseline entry.

  The real fix is upstream: either the Pest larastan plugin needs
  to narrow `expect()`'s return type generic from `?string` /
  `mixed` to `string`, OR each test file needs `Expectation::class`
  to be re-typed locally. Both are out of scope for an in-package
  baseline-shrink sub-batch. Defer this to a v0.3.x explicit
  upstream-coordination task.
- **PR-α-resid-2** — Fixture `@method static` annotations on
  AbstractEnumeratorClass tests (~50 entries). Still viable; not
  attempted in this pass.
- **PR-α-resid-3** — Trait composition `@mixin` audit. Annotate
  `Concerns/HasEnumAttributes`, `Concerns/RendersHtml`,
  `Concerns/HasInvokableCases` with `@mixin` directives that document
  the trait's host-type expectation; then the
  `function.alreadyNarrowedType` defensive checks PHPStan flags can
  be replaced with `assert()` or proper typing (~210 entries). Still
  viable; not attempted in this pass.

Even after PR-α-resid-2..3 the baseline is unlikely to clear the
flip-readiness threshold (≤ 300 patterns) on its own — PR-α-resid-1's
~150 entries are blocking. Path to the flip requires either the
upstream Pest narrowing OR a Pest-Mixins-Expectation overlay stub
that surgically fixes the `Expectation<T>::$not` resolution.

## Suggested ordering

For one cohesive v0.3.0 ship cycle, in order of dependency:

```
PR-α (baseline shrink, foundation)
   ↓
PR-β (coverage push)
   ↓
PR-γ (per-input wireModel)   ← natural follow-up to PR-F
PR-δ (multi-select listbox)  ← natural follow-up to PR-G
   ↓
PR-ε (coverage / mutation reporting)
   ↓
Release prep + tag v0.3.0
```

**PR-ζ + PR-η + PR-θ are nice-to-have.** If two of them slip to
v0.3.1 / v0.4.0, the v0.3.0 cycle is still a clean release.

## Open scope questions for the v0.3.0 audit-remediate pass

1. **Coverage target for v0.3.0 — 85% or 90%?** The current trajectory
   says 85% is reachable in 2-3 batches; 90% takes 4-5 and starts to
   feel like coverage-cargo-culting (the original assessment in
   [NEXT-SESSION.md](../_private/NEXT-SESSION.md) "The hard
   remaining 90% gap"). Decide before opening PR-β.

2. **Mutation testing on day 1, or after baseline?** `infection`
   integration matures across versions; a first-pass setup before
   PR-α's baseline shrink might noise-up the picture. After PR-α is
   cleaner.

3. **Browser-level CI test (PR-θ) — Playwright or Cypress?** Playwright
   has lighter CI overhead but a steeper Laravel-integration story.
   Cypress is heavier but better-documented for Laravel apps. Both
   add a `node_modules/` to the test rig. Worth a dedicated ADR.

4. **Continuous-deployment to Packagist via tag push?** Currently the
   Packagist hook would need to be wired manually (still in
   `NEXT-SESSION.md` "Manual external steps"). Once registered, every
   tag push auto-updates Packagist. Worth confirming before v0.3.0
   tag.
